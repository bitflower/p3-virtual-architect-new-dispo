# Assertion Model — How the Flow Tracker Generates Test Expectations

> **The user's continuation IS the confirmation.** The recorder just makes that implicit signal explicit by diffing what changed between actions. The user never has to think "now I should assert something" — their natural flow behavior is the spec.

## The cycle

```
USER ACTION → SYSTEM RESPONSE → USER CONTINUES → (implicit confirmation)
     │              │                   │
     │              │                   └─ By acting on newly appeared elements,
     │              │                      the user confirms the previous action
     │              │                      produced the expected result.
     │              │
     │              └─ New elements appear in the DOM.
     │                 Recorder snapshots the diff:
     │                 "these data-testid elements are new."
     │
     └─ Click, drag, select, fill.
        Recorder captures the action.
```

Each round:

1. **The user initiates an action** — clicks a filter, drags a card, confirms a dialog.

2. **The system responds** — new elements appear (lot cards after filtering, a dialog after dragging, shipment cards after selecting a lot). The recorder detects these via snapshot diff: it compares visible `data-testid` elements before and after the action settles.

3. **The user continues with the next step** — by acting on a newly appeared element, the user implicitly confirms that the previous action was successful. If lot cards didn't appear after the filter, the user wouldn't be clicking one. If the dialog didn't open after the drag, the user wouldn't be filling in a date.

4. **The cycle repeats** until the flow ends (the user clicks Stop).

## What the recorder does with this

The snapshot diff between rounds produces `expect-visible` assertions tagged `autoGenerated: true`. These make the implicit confirmation **explicit** in the generated test:

| User behavior | Recorder captures | Generator emits |
|---|---|---|
| Clicks HL filter, then clicks a lot card | Lot cards appeared between those two actions | `expect(lotCards.first()).toBeVisible()` |
| Drags shipment, then clicks date input | Dialog appeared between those two actions | `expect(createToDialog).toBeVisible()` |
| Clicks confirm, then stops recording | Spinner appeared and disappeared | `expect(spinner).toBeHidden()` |

## Why this works without explicit assertions

The user never thinks about assertions. They just **do the flow**. Their natural continuation behavior is the signal:

- **Continuing = success.** The user moved on, so the previous step worked.
- **Stopping or hesitating = potential failure.** If the user stops the recording mid-flow, the test ends there — which is correct, because the flow was incomplete.
- **Acting on a wrong element = test still valid.** The test captures what happened, not what should have happened. If the test later fails because the wrong thing appeared, that's the assertion doing its job.

## V2: Explicit assertions

V1 relies entirely on implicit confirmation via snapshot diff. V2 could add:

- **PO-initiated assertions** — right-click an element, choose "this should say X" → emits `expect-text` without `autoGenerated` flag
- **Count assertions** — "there should be exactly 3 lot cards" → emits `expect-count`
- **Absence assertions** — "this element should NOT be visible" → emits `expect-hidden`

These would be user-intentional, not auto-generated, so the generator treats them as non-negotiable.
